# Embedding - An embedding API/Worker

A monorepo containing a full search engine pipeline built on a web crawler, vector embeddings, and retrieval-augmented generation.

```
/
└── app/
	├──	embedding/          # Embedding API + RabbitMQ worker
	└── evaluation/        	# Evaluation package
```

---

## Architecture overview

```
Go Crawler
    │ publishes page_uuid
    ▼
RabbitMQ
    │
    ▼
embedding/worker
    │  fetch unembedded chunks from Postgres
    │  embed via Ollama (bge-m3 on Mac)
    │  save vectors back to Postgres
    │
embedding/api
    │  POST /api/search/stream
    │  1. expand query via LLM
    │  2. multi-query vector search (pgvector)
    │  3. stream summary via Ollama (llama3)
    │
evaluation/
    │  retrieval eval: Recall@K, MRR
    └─ RAG eval: RAGAS faithfulness / relevancy / precision / recall
```

---

## Prerequisites

| Service               | Where              | Notes                     |
| --------------------- | ------------------ | ------------------------- |
| PostgreSQL + pgvector | any                | `CREATE EXTENSION vector` |
| RabbitMQ              | any                | default guest/guest       |
| Ollama                | Mac (or local)     | serves bge-m3 + llama3.2  |
| Python 3.11+          | Windows (or local) | runs embedding + eval     |

## Quick start

### 1. Clone and configure

```bash
git clone https://github.com/Siripassornbibi/CP465-Text-Mining---AI-Search-Engine.git embedding
cd embedding
cp config.example.json config.json   # edit with your IPs and credentials
```

### 2. Start Ollama

```bash
ollama pull llama3.2
ollama pull bge-m3
ollama serve
```

Note: Set `OLLAMA_HOST` environment variable to `0.0.0.0` to allows external request

### 3. Install Python dependencies

```bash
# root directory
pip install -e .

# embedding package
cd app/embedding
pip install -r requirements.txt

# eval package
cd app/evaluation
pip install -r requirements.txt
```

### 4. Create a `.env` file

```env
DB_URL=postgresql://user:pass@localhost/googai
CONFIG_PATH=config.json
```

### 5. Start the API

```bash
python -m embedding
```

### 6. Start the worker (separate terminal)

```bash
python -m embedding worker
```

## Configuration

All runtime settings live in `config.json`. The database URL is read from the `DB_URL` environment variable and must never appear in `config.json`.

```json
{
    "rabbitmq": {
        "url": "amqp://guest:guest@192.168.1.50:5672/",
        "exchange": "prawler",
        "exchange_type": "topic",
        "queue": "prawler.embedding",
        "routing_key": "prawler.embedding.#",
        "prefetch_count": 4
    },
    "embedder": {
        "backend": "ollama",
        "model": "bge-m3",
        "ollama_url": "http://192.168.1.50:11434",
        "timeout": 60.0
    },
    "llm": {
        "model": "llama3.2",
        "ollama_url": "http://192.168.1.50:11434"
    },
    "api": {
        "host": "0.0.0.0",
        "port": 8000,
        "top_k": 8,
        "expand_queries": 2
    }
}
```

### Key settings

| Key                       | Description                                                                  |
| ------------------------- | ---------------------------------------------------------------------------- |
| `embedder.backend`        | `"ollama"` (remote Mac) or `"local"` (sentence-transformers on this machine) |
| `embedder.model`          | Model name as shown by `ollama list` — must match exactly                    |
| `api.top_k`               | Number of chunks retrieved per query                                         |
| `api.expand_queries`      | Extra queries generated by LLM for retrieval (`0` = disabled)                |
| `rabbitmq.prefetch_count` | Max concurrent messages per worker process                                   |

### Embedder backends

**Remote Ollama:**

```json
"embedder": { "backend": "ollama", "model": "bge-m3", "ollama_url": "http://192.168.1.50:11434" }
```

**Local sentence-transformers:**

```json
"embedder": { "backend": "local", "model": "BAAI/bge-m3", "device": null, "batch_size": 32 }
```

### API endpoints

| Method | Path                 | Description                                         |
| ------ | -------------------- | --------------------------------------------------- |
| `GET`  | `/api/health`        | Health check — returns model names and queue config |
| `POST` | `/api/search`        | Full JSON response with summary + sources           |
| `POST` | `/api/search/stream` | SSE stream — sources first, then summary tokens     |
| `GET`  | `/`                  | React frontend (if `search-ui/dist/` exists)        |

### Search stream event format

```
event: queries
data: ["rephrased query 1", "rephrased query 2"]

event: sources
data: [{"title": "...", "url": "...", "score": 0.91}, ...]

data: "summary token by token..."

data: "[DONE]"
```

## Running evaluation

```bash
# retrieval only — fast, no LLM needed
python -m evaluation

# full RAG eval with RAGAS — uses Ollama LLM
python -m evaluation --rag

# custom options
python -m evaluation --cases path/to/test_cases.json --top-k 10
```

### `test_cases.json` format

```json
{
    "How does Go handle concurrency?": {
        "expected_urls": ["https://go.dev/doc/effective_go"],
        "ground_truth": "Go uses goroutines and channels..."
    }
}
```

### Retrieval metrics

| Metric   | Poor  | Acceptable | Good  |
| -------- | ----- | ---------- | ----- |
| Recall@1 | < 40% | 40–60%     | > 70% |
| Recall@5 | < 60% | 60–80%     | > 85% |
| MRR      | < 0.4 | 0.4–0.6    | > 0.7 |

### RAGAS metrics (0–1, higher is better)

| Metric              | Measures                                           |
| ------------------- | -------------------------------------------------- |
| `faithfulness`      | Does the answer stay within the retrieved context? |
| `answer_relevancy`  | Does the answer address the question?              |
| `context_precision` | Are the retrieved chunks relevant?                 |
| `context_recall`    | Did retrieval find all necessary chunks?           |

## Embedding versions

The `chunks.embedding_version` column tracks which embedding format was used. Bumping `CURRENT_EMBEDDING_VERSION` in `ports/chunk_repository.py` causes the worker to automatically re-embed outdated chunks on the next publish cycle — no manual SQL needed.

| Version | Description                                                    |
| ------- | -------------------------------------------------------------- |
| 1       | No prefix, no page metadata                                    |
| 2       | `PASSAGE_PREFIX` + `page_title` + `page_description` + content |

To re-embed everything after a model or format change:

```bash
# queue all pages again via the Go publisher
LOOP=false go run publisher.go
```

---

## Query expansion

When `api.expand_queries > 0`, the API generates additional query phrasings using the LLM before retrieval. All queries are embedded and retrieved in parallel, results are merged by chunk identity and re-ranked by best score.

Set `expand_queries: 0` in `config.json` to disable and use only the original query. This is useful for A/B testing whether expansion improves your recall scores.
